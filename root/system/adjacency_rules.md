# Adjacency Rules

## 1. Adjacency is structural, not semantic
Adjacency defines:
- what can be next to what  
- what cannot be next to what  

Adjacency does not define:
- meaning  
- identity  
- semantics  
- interpretation  

Adjacency is non‑semantic.

## 2. Adjacency is upstream of contamination
Adjacency establishes:
- allowed nearness  
- forbidden nearness  

Contamination describes what happens when forbidden adjacency occurs.  
Adjacency comes first.

## 3. Adjacency cannot assign identity
Adjacency does not:
- define what something is  
- define what something means  
- define category  
- define semantics  

Adjacency is not naming.

## 4. Adjacency is constrained by geometry
Adjacency must obey:
- shape boundaries  
- dimensional constraints  
- geometric forms  

Adjacency cannot violate geometry.

## 5. Adjacency prevents fusion
Adjacency ensures:
- entities do not merge  
- domains do not fuse  
- boundaries remain intact  

Adjacency is a membrane.

## 6. Adjacency is reversible
Adjacency actions:
- do not mutate identity  
- do not mutate semantics  
- do not mutate structure  

Adjacency leaves no residue.

## 7. Adjacency cannot create new domains
Adjacency cannot:
- create new categories  
- create new semantics  
- create new containers  
- create new routing paths  

Adjacency only uses existing structure.

## 8. Adjacency is explicit, not implicit
Adjacency must be:
- declared  
- bounded  
- intentional  

Implicit adjacency leads to drift.

## 9. Adjacency prevents placement inversion
Adjacency ensures:
- placement remains placement  
- placement does not become meaning  
- placement does not become identity  

Adjacency protects placement from semantic drift.

## 10. Adjacency must remain isolated
Adjacency must not fuse with:
- geometry  
- placement  
- routing  
- semantics  
- registry  
- engine logic  

Adjacency is a standalone conceptual domain.
