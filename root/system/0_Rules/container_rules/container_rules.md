# Container Rules

## 1. Containers are structural, not semantic
Containers define:
- what can hold something  
- what cannot hold something  

Containers do not define:
- meaning  
- identity  
- semantics  
- interpretation  

Containers are non‑semantic.

## 2. Containers cannot assign identity
Containers do not:
- define what something is  
- define what something means  
- define category  
- define semantics  

Containers are not naming.

## 3. Containers are constrained by geometry
Containers must obey:
- shape boundaries  
- dimensional constraints  
- geometric forms  

Containers cannot violate geometry.

## 4. Containers are upstream of placement
Containers define:
- what can be placed inside  
- what cannot be placed inside  

Placement defines:
- where containers go  
- where contents go  

Containers come first.

## 5. Containers prevent contamination
Containers ensure:
- contents do not leak  
- domains do not fuse  
- boundaries remain intact  

Containers are membranes.

## 6. Containers are reversible
Container operations:
- do not mutate identity  
- do not mutate semantics  
- do not mutate structure  

Containers leave no residue.

## 7. Containers cannot create new domains
Containers cannot:
- create new categories  
- create new semantics  
- create new containers  
- create new routing paths  

Containers only use existing structure.

## 8. Containers are explicit, not implicit
Containers must be:
- declared  
- bounded  
- intentional  

Implicit containers lead to drift.

## 9. Containers cannot interpret behavior
Containers do not:
- infer intent  
- interpret meaning  
- evaluate content  

Containers are structural, not cognitive.

## 10. Containers must remain isolated
Containers must not fuse with:
- geometry  
- placement  
- adjacency  
- routing  
- semantics  
- registry  
- engine logic  

Containers are a standalone conceptual domain.
