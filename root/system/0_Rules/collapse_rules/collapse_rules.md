# Collapse Rules

## 1. Collapse is a system state, not an object
Collapse is not a container, coordinate, function, category, or structural element.  
It is a **state** the system enters when invariants fail.

## 2. Collapse has no name, identity, or coordinates
Collapse cannot be:
- named  
- placed  
- routed to  
- referenced as a container  
- assigned meaning  

Collapse is not a structural entity.

## 3. Collapse is triggered by invariant violation
Collapse occurs when one or more of the following fail:
- adjacency rules  
- contamination rules  
- naming rules  
- category boundaries  
- placement rules  
- drift thresholds  
- registry/engine separation  
- structure/engine separation  
- coordinate/meaning separation  

Collapse is always the result of **invariant failure**, not system intent.

## 4. Collapse is prevented by membranes
The following membranes prevent collapse:
- category boundaries  
- contamination boundaries  
- naming boundaries  
- adjacency boundaries  
- placement boundaries  
- drift detection boundaries  

Membranes are conceptual constraints that maintain system integrity.

## 5. Collapse is detected early
Early collapse indicators include:
- adjacency distortion  
- naming inversion  
- coordinate misalignment  
- routing misinterpretation  
- contamination signals  
- drift accumulation  

Detection is structural, not semantic.

## 6. Collapse is upstream of engine execution
The engine:
- cannot run collapse  
- cannot interpret collapse  
- cannot route into collapse  
- cannot execute collapse  

Collapse is resolved **before** engine execution.

## 7. Collapse cannot propagate across domains
Collapse in one domain (structure, registry, semantics, engine, geometry, governance)  
must remain isolated and cannot spread to other domains.

## 8. Collapse recovery is structural, not semantic
Recovery actions include:
- resetting coordinates  
- re‑asserting boundaries  
- re‑asserting naming rules  
- re‑asserting adjacency  
- re‑asserting placement  

Recovery does **not** involve:
- renaming containers  
- reinterpreting meaning  
- rewriting semantics  

## 9. Collapse cannot be used as a mechanism
Collapse is not:
- a tool  
- a feature  
- a routing mechanism  
- a governance action  

Collapse is only a **failure state**.

## 10. Collapse must be logged but not interpreted
The system:
- logs collapse  
- does not interpret collapse  
- does not assign meaning  
- does not route based on collapse  

Collapse is recorded, not understood.
