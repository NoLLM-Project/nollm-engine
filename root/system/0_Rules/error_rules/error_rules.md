# Error Rules

## 1. Errors are structural, not semantic
An error is a structural mismatch or violation.  
Errors arise from:
- shape mismatches  
- boundary violations  
- invalid adjacency  
- invalid placement  
- category mismatches  
- wrapper mismatches  

Errors do not arise from:
- meaning  
- naming interpretation  
- semantic disagreement  

Errors are mechanical facts.

## 2. Errors occur upstream of execution
Errors are detected:
- before routing  
- before execution  
- before engine movement  

The engine does not discover errors; it receives them.

## 3. Errors do not imply collapse
Errors are:
- local  
- bounded  
- reversible  

Errors do not:
- propagate  
- contaminate  
- invert domains  
- cause collapse  

Errors are not systemic failures.

## 4. Errors are boundary violations
Errors occur when:
- a domain crosses a boundary  
- a shape does not match  
- a wrapper is missing  
- a category is misapplied  
- a membrane is breached  

Errors are boundary‑based, not semantic.

## 5. Errors are reversible
Error recovery involves:
- restoring boundaries  
- correcting shapes  
- re‑asserting categories  
- re‑establishing placement  

Recovery does not involve:
- renaming  
- reinterpreting  
- rewriting semantics  

Errors leave no residue.

## 6. Errors cannot create new pathways
Errors cannot:
- create new adjacency  
- create new routing  
- create new categories  
- create new semantics  
- create new containers  

Errors only indicate invalid use of existing pathways.

## 7. Errors are non‑interpretive
Errors do not:
- interpret meaning  
- infer intent  
- assign semantics  

Errors are structural facts.

## 8. Errors must be logged, not interpreted
The system:
- logs errors  
- does not interpret them  
- does not assign meaning  
- does not route based on them  

Errors are recorded, not understood.

## 9. Errors cannot be used as mechanisms
Errors are not:
- tools  
- features  
- routing mechanisms  
- governance actions  

Errors are only signals.

## 10. Errors must remain isolated
Errors in:
- structure  
- registry  
- semantics  
- engine  
- geometry  
- governance  

must remain local and cannot spread.
