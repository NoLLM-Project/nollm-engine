# Registry Rules

## 1. Registry is mechanical, not semantic
Registry defines:
- how labels map to references  
- how references map to containers  

Registry does not define:
- meaning  
- identity  
- semantics  
- interpretation  

Registry is non‑semantic.

## 2. Registry is downstream of naming
Naming defines:
- the label  

Registry defines:
- the mapping  

Registry cannot override naming.

## 3. Registry cannot assign identity
Registry does not:
- define what something is  
- define what something means  
- define category  
- define semantics  

Registry is not identity.

## 4. Registry is constrained by invariants
Registry must obey:
- system invariants  
- governance invariants  
- non‑mutation  
- non‑fusion  
- non‑inversion  

Registry cannot violate invariants.

## 5. Registry prevents semantic drift
Registry ensures:
- mappings do not imply meaning  
- mappings do not imply category  
- mappings do not imply identity  

Registry is a membrane.

## 6. Registry is reversible
Registry operations:
- do not mutate identity  
- do not mutate semantics  
- do not mutate structure  

Registry leaves no residue.

## 7. Registry cannot create new domains
Registry cannot:
- create new categories  
- create new semantics  
- create new containers  
- create new routing paths  

Registry only maps what exists.

## 8. Registry must be explicit
Registry must be:
- declared  
- bounded  
- intentional  

Implicit registry leads to drift.

## 9. Registry cannot interpret behavior
Registry does not:
- infer intent  
- interpret meaning  
- evaluate content  

Registry is mechanical, not cognitive.

## 10. Registry must remain isolated
Registry must not fuse with:
- naming  
- semantics  
- identity  
- placement  
- adjacency  
- routing  
- engine logic  

Registry is a standalone conceptual domain.
