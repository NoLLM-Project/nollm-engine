# Coordination Rules

## 1. Coordination is relational, not hierarchical
Coordination defines how domains relate without establishing control, dominance, or interpretation.  
No domain commands another.  
No domain becomes another.

## 2. Coordination preserves separation
Coordination must never cause:
- fusion  
- inversion  
- contamination  
- semantic bleed  
- structural mutation  

Coordination is a membrane, not a merger.

## 3. Coordination occurs through interfaces, not meaning
Domains coordinate through:
- shapes  
- wrappers  
- boundaries  
- contracts  

Coordination does not use:
- semantics  
- naming  
- interpretation  
- inference  

Coordination is mechanical, not semantic.

## 4. Coordination is upstream of execution
Coordination defines:
- what can talk  
- what cannot talk  
- what can pass  
- what cannot pass  

These constraints exist **before** engine execution.  
The engine does not coordinate; it executes.

## 5. Coordination is shape‑based, not name‑based
Coordination relies on:
- structural shapes  
- interface forms  
- wrapper boundaries  

It does not rely on:
- container names  
- function names  
- semantic labels  

Naming does not drive coordination.

## 6. Coordination is non‑interpretive
No domain interprets:
- content  
- meaning  
- naming  
- semantics  

Coordination is non‑semantic and non‑inferential.

## 7. Coordination prevents cross‑domain drift
Coordination ensures:
- structure stays structure  
- engine stays engine  
- registry stays registry  
- semantics stays semantics  
- content stays content  

No domain drifts into another.

## 8. Coordination is reversible
Coordination actions:
- do not mutate identity  
- do not mutate structure  
- do not mutate meaning  
- do not mutate registry  

Coordination leaves no residue.

## 9. Coordination is explicit, not implicit
Coordination must be:
- declared  
- bounded  
- intentional  

Implicit coordination leads to drift.

## 10. Coordination cannot create new pathways
Coordination cannot:
- create new adjacency  
- create new routing  
- create new categories  
- create new semantics  
- create new containers  

Coordination only uses pathways that already exist.
