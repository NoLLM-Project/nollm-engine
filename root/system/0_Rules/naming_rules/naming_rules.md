# Naming Rules

## 1. Names are labels, not meaning
Names define:
- how something is referred to  
- how something is addressed  

Names do not define:
- what something is  
- what something means  
- how something behaves  
- what category it belongs to  

Names are non‑semantic.

## 2. Naming cannot assign identity
Naming does not:
- define identity  
- define category  
- define semantics  
- define ontology  

Naming is not identity.

## 3. Naming is constrained by invariants
Naming must obey:
- system invariants  
- governance invariants  
- non‑mutation  
- non‑fusion  
- non‑inversion  

Naming cannot violate invariants.

## 4. Naming is upstream of registry
Naming defines:
- the label  

Registry defines:
- the mapping  

Naming comes first.

## 5. Naming prevents semantic drift
Naming ensures:
- labels do not imply meaning  
- labels do not imply category  
- labels do not imply identity  

Naming is a membrane.

## 6. Naming is reversible
Naming actions:
- do not mutate identity  
- do not mutate semantics  
- do not mutate structure  

Naming leaves no residue.

## 7. Naming cannot create new domains
Naming cannot:
- create new categories  
- create new semantics  
- create new containers  
- create new routing paths  

Naming only labels what exists.

## 8. Naming must be explicit
Naming must be:
- declared  
- bounded  
- intentional  

Implicit naming leads to drift.

## 9. Naming cannot interpret behavior
Naming does not:
- infer intent  
- interpret meaning  
- evaluate content  

Naming is referential, not cognitive.

## 10. Naming must remain isolated
Naming must not fuse with:
- semantics  
- identity  
- registry  
- placement  
- adjacency  
- routing  
- engine logic  

Naming is a standalone conceptual domain.
