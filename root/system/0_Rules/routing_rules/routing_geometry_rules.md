# Routing Geometry Rules

Routing geometry rules define how routing may use geometric structure without accessing, interpreting, or mutating geometry definitions.  
Geometry is upstream; routing is downstream.  
Routing may only consume geometric consequences expressed through adjacency and placement.

Routing is mechanical, non‑semantic, and non‑interpretive.

---

# 1. Geometry Visibility Rules

## 1.1 Routing cannot see geometry definitions
Routing may not access:
- geometric construction rules  
- shape definitions  
- boundary definitions  
- dimensional rules  

Geometry is upstream and invisible.

## 1.2 Routing may only see geometry through structural surfaces
Routing may access:
- adjacency  
- placement  
- coordinate relationships  

These are geometric consequences, not geometry itself.

---

# 2. Geometry Interpretation Rules

## 2.1 Routing cannot interpret geometry
Routing cannot:
- infer geometric meaning  
- reinterpret shapes  
- reinterpret boundaries  
- reinterpret dimensionality  

Routing uses geometry mechanically.

## 2.2 Routing cannot reinterpret structural geometry expressions
Routing cannot reinterpret:
- adjacency as meaning  
- placement as semantics  
- coordinate relationships as categories  

Structure is taken as‑is.

---

# 3. Geometry Mutation Rules

## 3.1 Routing cannot mutate geometry
Routing cannot:
- change shapes  
- change boundaries  
- change dimensions  
- change geometric rules  

Geometry is immutable and upstream.

## 3.2 Routing cannot create geometric constructs
Routing cannot create:
- new shapes  
- new boundaries  
- new adjacency types  
- new placement types  

Routing is not generative.

---

# 4. Geometry Boundary Rules

## 4.1 Routing cannot cross geometric domain boundaries
Routing cannot enter:
- geometry definitions  
- geometry invariants  
- geometry construction logic  

Routing stays within routing.

## 4.2 Routing cannot fuse with geometry
Routing cannot become:
- geometry  
- structure  
- naming  
- semantics  

Fusion is contamination.

---

# 5. Geometry‑Driven Movement Rules

## 5.1 Routing must respect geometric boundaries
Routing must obey:
- shape boundaries  
- container boundaries  
- dimensional limits  

These are enforced through adjacency and placement.

## 5.2 Routing cannot bypass geometric constraints
Routing cannot:
- move across non‑adjacent boundaries  
- violate placement rules  
- traverse forbidden geometric regions  

Geometry constrains routing.

---

# 6. Geometry‑Derived Transition Rules

## 6.1 Routing may only use transitions permitted by geometry
Allowed transitions:
- adjacency‑based movement  
- placement‑based movement  
- coordinate‑valid movement  

## 6.2 Routing cannot invent geometry‑derived transitions
Routing cannot:
- create new adjacency  
- create new placement  
- create new coordinate relationships  

Transitions are upstream.

---

# 7. Geometry Error Rules

## 7.1 Routing may detect geometry violations only through structural surfaces
Allowed geometry‑derived errors:
- invalid adjacency  
- invalid placement  
- boundary violations  

These are structural expressions of geometry.

## 7.2 Routing may not detect geometry semantics
Forbidden:
- shape meaning errors  
- dimensional interpretation errors  
- geometric intent errors  

Geometry semantics are upstream.

---

# 8. Geometry Stability Rules

## 8.1 Geometry is stable and non‑drifting
Routing must treat geometry as:
- fixed  
- immutable  
- non‑evolving  

## 8.2 Routing cannot drift in response to geometry
Routing cannot:
- reinterpret geometry over time  
- accumulate geometric meaning  
- adapt routing behavior based on geometry semantics  

Routing is stable.
