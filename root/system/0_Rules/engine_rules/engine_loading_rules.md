# Engine Loading Rules

Engine loading rules define how the engine is initialized, how it receives structural inputs, and what it is forbidden to do during loading.  
Loading is mechanical, non‑semantic, non‑interpretive, and strictly downstream of all invariants and membranes.

The engine may only load what it is permitted to see.  
The engine may not interpret, mutate, or influence upstream physics during loading.

---

# 1. Loading Scope Rules

## 1.1 Loading is mechanical
During loading, the engine may:
- receive structural references  
- receive adjacency tables  
- receive placement tables  
- receive routing tables  
- receive workflow sequences  

These are mechanical and non‑semantic.

## 1.2 Loading cannot expand engine capability
Loading cannot:
- grant new visibility  
- grant new execution powers  
- grant new state types  
- grant new domain access  

Loading does not modify the engine’s nature.

---

# 2. Loading Interpretation Rules

## 2.1 Loading cannot interpret meaning
The engine may not:
- infer intent  
- assign semantics  
- reinterpret names  
- reinterpret content  

Loading is non‑semantic.

## 2.2 Loading cannot reinterpret structure
The engine may not:
- reinterpret adjacency  
- reinterpret placement  
- reinterpret geometry  
- reinterpret boundaries  

Structure is accepted as‑is.

---

# 3. Loading Mutation Rules

## 3.1 Loading cannot mutate upstream domains
Loading cannot change:
- structure  
- geometry  
- naming  
- categories  
- invariants  

Loading is downstream and non‑mutative.

## 3.2 Loading cannot create new domains
Loading cannot create:
- new containers  
- new categories  
- new routing paths  
- new semantics  

Loading is not generative.

---

# 4. Loading Boundary Rules

## 4.1 Loading cannot cross domain boundaries
Loading cannot access:
- semantics  
- registry internals  
- naming rules  
- geometry definitions  
- invariants  

Loading stays within the engine domain.

## 4.2 Loading cannot fuse with other domains
Loading cannot become:
- structure  
- registry  
- semantics  
- content  

Fusion is contamination.

---

# 5. Loading Validation Rules

## 5.1 Loading may only validate structural integrity
Allowed validations:
- adjacency consistency  
- placement consistency  
- routing table consistency  
- workflow sequence consistency  

These are mechanical checks.

## 5.2 Loading may not validate semantic correctness
Forbidden validations:
- meaning correctness  
- identity correctness  
- category correctness  
- content correctness  

Semantic validation is upstream.

---

# 6. Loading State Rules

## 6.1 Loading state must be ephemeral
Loading state must:
- exist only during initialization  
- be cleared after loading  
- leave no residue  

## 6.2 Loading state cannot persist across executions
Loading state cannot:
- accumulate  
- evolve  
- adapt  
- drift  

Persistence is forbidden.

---

# 7. Loading Error Rules

## 7.1 Loading may only detect structural loading errors
Allowed:
- malformed adjacency  
- malformed placement  
- malformed routing tables  
- malformed workflow sequences  

## 7.2 Loading may not detect semantic errors
Forbidden:
- meaning errors  
- identity errors  
- category errors  
- content errors  

Semantic errors are upstream.

---

# 8. Loading Stability Rules

## 8.1 Loading must be deterministic
Given the same:
- adjacency  
- placement  
- routing  
- workflow  

Loading must produce the same engine state.

## 8.2 Loading cannot drift
Loading cannot:
- reinterpret itself  
- accumulate meaning  
- change behavior over time  

Loading is stable.
