# Geometry Rules

## 1. Geometry is structural shape, not meaning
Geometry defines:
- shapes  
- forms  
- boundaries  
- dimensions  

Geometry does not define:
- semantics  
- identity  
- naming  
- meaning  

Geometry is non‑semantic.

## 2. Geometry is upstream of adjacency
Geometry establishes:
- what shapes exist  
- how shapes relate  
- what boundaries exist  

Adjacency determines which shapes may touch.  
Geometry comes first.

## 3. Geometry is non‑interpretive
Geometry does not:
- interpret meaning  
- infer intent  
- assign semantics  
- encode identity  

Geometry is purely structural.

## 4. Geometry defines allowable forms
Geometry defines:
- valid shapes  
- valid boundaries  
- valid dimensional relationships  

Geometry does not define:
- placement  
- routing  
- category  
- naming  

Geometry is form, not function.

## 5. Geometry prevents shape drift
Geometry ensures:
- shapes remain stable  
- boundaries remain stable  
- dimensions remain stable  

Shape drift is a geometry violation.

## 6. Geometry is independent of containers
Geometry does not depend on:
- container names  
- container semantics  
- container identity  

Containers use geometry; geometry does not use containers.

## 7. Geometry cannot be mutated by execution
The engine cannot:
- change shapes  
- change boundaries  
- change dimensions  

Execution does not mutate geometry.

## 8. Geometry is reversible
Geometry operations:
- do not mutate identity  
- do not mutate semantics  
- do not mutate registry  

Geometry leaves no residue.

## 9. Geometry cannot create new domains
Geometry cannot:
- create new categories  
- create new semantics  
- create new containers  
- create new routing paths  

Geometry only defines shape.

## 10. Geometry must remain isolated
Geometry must not fuse with:
- adjacency  
- placement  
- routing  
- semantics  
- registry  
- engine logic  

Geometry is a standalone conceptual domain.
